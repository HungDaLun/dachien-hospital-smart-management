# Part 1: 知識架構師的心法 (Methodology)

## 1. 核心觀念：格式決定智商
AI 並不是像人類一樣「閱讀」文件，它是在計算數學向量。
*   **雜訊 (Noise)**：Word 的排版、Excel 的空白列，這些對人類友善的格式，對 AI 來說都是干擾。
*   **結構 (Structure)**：只有結構乾淨的資料，才能建立精準的索引。

**架構師的第一任務：清除雜訊，建立結構。**

## 2. 三大核心格式標準
為了讓 AI 的理解力最大化，請不再使用為了「列印」而生的格式，改用「運算」導向的格式：

| 檔案類型 | 推薦 AI 格式 | 適用場景 | 轉換原則 |
| :--- | :--- | :--- | :--- |
| **文件類** | **Markdown (.md)** | 規章、手冊、合約 | **層級分明**：用 `#` 標示標題，去除頁眉頁腳雜訊。 |
| **數據類 (小)** | **JSONL (.jsonl)** | 產品表、對話紀錄 | **扁平化**：將 Excel 多分頁合併，每一行都是獨立物件。 |
| **數據類 (大)** | **Parquet** | 交易日誌、Log | **效能優先**：適合 TB 級別大規模訓練。 |

## 3. 資料處理 SOP (標準作業程序)
1.  **盤點 (Audit)**：區分文件類與數據類資料。
2.  **清洗 (Clean)**：去個資 (PII Scrubbing)、刪除過期文件。
3.  **轉換 (Convert)**：執行 Markdown/JSONL 格式轉換。
4.  **上傳 (Ingest)**：送入知識庫。

## 4. 企業知識治理標準 (EAKAP Governance v1.0)
這不只是檔名問題，而是未來 Agent 能否精準調用知識的關鍵。採用「圖書館模型」：

*   **檔名 (Filename) = 書背上的索書號**
    *   **目的**：給人類看、在檔案總管排序、避免重名。
    *   **原則**：剛性、簡短、標準化。
    *   **格式**：`[歸屬權]-[文件類別]-[核心主題]-[屬性後綴].ext`
        *   `[歸屬權]`: **必須**存在於系統部門列表 (如 `HR`, `IT`, `SALES`)。禁止 AI 自創 `REG` 等幽靈部門。
        *   `[文件類別]`: **必須**存在於系統文件類別列表 (如 `Policy`, `Spec`).
        *   `[核心主題]`: 英文/數字，PascalCase (如 `RemoteWork`)。
        *   `[屬性後綴]`: 版本或日期 (如 `v2024`)。
    *   **範例**：`HR-Policy-RemoteWork-v2024.pdf`

*   **元資料 (Metadata) = 數位檢索卡**
    *   **目的**：給 AI (Agent) 看、或是做進階篩選。
    *   **原則**：彈性、豐富、多維度。
    *   **必要維度**：
        1.  **Product Context**: `["Origins", "Mega-Mushroom"]`
        2.  **Target Audience**: `["Internal", "Public"]`
        3.  **Governance Status**: `["Draft", "Approved"]`

# Part 2: 平台實踐的劍法 (Platform Implementation)

理解了上述的資料處理原則後，我們不能依賴 (員工手動上傳檔案。我們需要一個**企業級 AI 知識庫平台 (EAKAP)** 來自動化與管理這一切。

## 1. 平台願景
建立一個「自動化知識運籌中心」，具備自動格式轉換、標準化命名、權限管理，且不被單一 AI 供應商綁架。

## 2. 核心架構：Hub & Spoke (軸輻式)
為了兼顧「Gemini 的強大運算力」與「企業資料主權」，我們採用雙層儲存策略：

### Layer 1: Hub (企業自有資產區)
*   **技術**：自建 S3 儲存桶 + PostgreSQL。
*   **定位**：**唯一的真理來源 (Source of Truth)**。所有原始文件 (PDF/Word) 經過 **AI 館長** 轉譯後的乾淨 Markdown 必須存這裡。
*   **優勢**：資料永遠在自己手上，未來想換 AI 模型時，原始檔都在。

### Layer 2: Spoke (AI 運算熱區)
*   **技術**：Gemini File Search API。
*   **定位**：**運算效能層**。
*   **運作**：系統透過 **Sync Adapter** 自動將 Hub 中的檔案同步給 Gemini。Agent 實際運作時，是讀取這一層的資料以獲得最快速度。

## 3. 關鍵功能設計
*   **Agent Factory (代理人工廠)**：
    *   **概念**：從「手動組裝 Agent」進化為「說出願望，工廠自動生產」。
    *   **運作**：使用者輸入需求（如「我要一個財務助理」），**Architect Agent** 會自動撈取 `Department=Finance` 的文件，並根據標準模版（如 K-0）撰寫系統指令，即時生成 Agent。
*   **Metadata Trinity (元數據鐵三角)**：
    *   **Who (人)**：綁定 User Profile，確立權責歸屬。
    *   **Where (組織)**：綁定 Department，建立資料圍籬 (Data Fencing)。
    *   **What (知識)**：綁定 Unified Taxonomy (統一分類表)，由 AI 預判分類，人類最終確認 (Human-in-the-Loop)。
    *   **Naming (命名)**：採用「圖書館索書號」邏輯 (`[部門]-[類型]-[主題]-[屬性]`)，AI 依據資料庫中有效的部門代碼自動生成，嚴禁自創代號 (如 `REG`)。
*   **Smart Upload (智慧上傳)**：
    *   **流程**：User 拖曳檔案 -> AI 預填 Metadata (部門、分類) -> User 點擊 One-click Confirm -> 進入 Hub。
    *   **價值**：確保進入系統的每一份資料都有完美的標籤，「垃圾不進，垃圾不出」。
*   **權限隔離 (RBAC)**：
    *   透過「標籤 (Tags)」管理知識邊界（如：`Tag=HR` 的文件只有 `HR_Agent` 能讀）。
    *   即使底層模型看過所有資料，應用層也會確保員工無法越權存取。
*   **強制引用 (Grounding)**：
    *   設定 Agent Skills，強制 Agent 在回答特定問題前，必須先調用 `consult_knowledge` 工具。
    *   系統指令 (System Instruction) 強制 Agent 必須引用知識庫內容，並顯示來源連結，杜絕 AI 幻覺。

# Part 3: DIKW 可視化大腦 (The Visualized Enterprise Brain)

這不只是一個管理後台，這是一個讓企業主能「看見」自己公司大腦正在生長的戰情室。我們將 **DIKW (Data, Information, Knowledge, Wisdom)** 架構具象化為互動式儀表板。

## 1. 核心體驗：點亮知識星系 (Light It Up)
系統首頁是一張動態的「知識星系圖」，由內而外分為四層：

1.  **D (Data) 核心層**：原始檔案區。初始狀態為暗淡。
2.  **I (Information) 萃取層**：從檔案中提取的事實（如：市場規模數字、競品定價表）。
3.  **K (Knowledge) 框架層**：SWOT、VPC、Persona 等 150+ 種知識框架節點。初始為灰色。
4.  **W (Wisdom) 賦能層**：最終可產出的戰略文件（如：募資簡報、危機公關稿）。

**互動流程 (The Feedback Loop)：**
1.  **上傳**：使用者拖曳一份「競品財報 PDF」進入 D 層。
2.  **消化**：D 層核心閃爍，光線向外射出。
3.  **提取**：光線觸發 I 層的「營收數據」、「產品規格」節點亮起。
4.  **歸納**：光線匯聚到 K 層的「SWOT 分析」節點，系統顯示「威脅 (Threats)」象限已被填滿，信心度 85%。
5.  **解鎖**：W 層的「競爭策略報告」顯示「可生成」，並提示「尚缺市場份額數據以解鎖完整版」。

## 2. 彈性架構：動態框架引擎 (Dynamic Framework Engine)

為了避免「窮舉所有框架」導致系統僵化，我們採用 **Metadata 驅動** 設計：

*   **拒絕寫死 (No Hard-Coding)**：工程師不寫死 SWOT 或 PESTLE 的欄位。
*   **積木式定義**：所有框架以 JSON 格式存於資料庫（例如定義 SWOT 由四個文字區塊組成）。
*   **AI 自動填空**：當檔案上傳時，**Knowledge Mapper Agent** 會自動掃描所有已定義的框架，嘗試將新資訊填入對應的 JSON 欄位中。

**優勢：**
*   **無限擴充**：顧問想新增一個「AI 準備度評估表」？只需在後台新增一筆 JSON 設定，前台星系圖就會自動長出這顆新星星。
*   **容錯性**：資料不全沒關係，系統會顯示「完整度 30%」，引導使用者繼續補充資料，而不是直接報錯。

# Part 4: 商業落地與交付模式 (Delivery Models)

除了雲端 SaaS 模式，本架構因「Hub (資料)」與「Spoke (運算)」解耦的特性，極度適合發展 **「AI Appliance (AI 一體機)」** 的硬體交付模式。

## 1. AI Appliance 產品概念 (The Mac Mini Strategy)
將整套 EAKAP 平台預先封裝於高效能迷你電腦（如 **Mac Mini**）中，打造成企業的「私有 AI 大腦主機」。

*   **產品型態**：軟硬整合一體機 (Software in a Box)。
*   **目標客群**：中小企業 (SME)、傳統產業、醫療診所、法律事務所。
*   **解決痛點**：
    1.  **訂閱疲勞**：老闆不喜歡每個月付錢，喜歡「買斷」資產的感覺。
    2.  **資料焦慮**：覺得資料上雲不安全，機器擺在辦公室看得到才安心。
    3.  **IT 門檻**：沒有工程師能設定 Docker/雲端環境。

## 2. 混合雲運作模式 (Hybrid-Cloud Operation)
*   **本地端 (On-Premise Device)**：
    *   **設備**：Mac Mini (M2/M4晶片, 16GB RAM, 512GB SSD)。
    *   **運行內容**：前端介面 (Next.js)、PostgreSQL 資料庫、MinIO 文件儲存 (Hub)。
    *   **意義**：所有原始 Knowledge Files 物理上儲存在這台機器裡，**不離開公司內網**。
*   **雲端 (Cloud API)**：
    *   **運行內容**：Gemini File Search API (Spoke)。
    *   **互動**：僅在「需要運算」時，由機器自動加密上傳資料至 Gemini 作為「運算快取」，運算完後資料依生命週期策略移除，原始檔始終保留在本地機器。

## 3. 交付與獲利流程
1.  **採購**：客戶下單一台「AI 知識庫主機」。
2.  **預載**：出廠前預先安裝好 Docker 環境與 EAKAP 系統圖像。
3.  **交付**：客戶收到機器，插上電源與網路線。
4.  **開通**：客戶在瀏覽器輸入 `http://ai-brain.local`，輸入 Gemini API Key 即開始運作。
5.  **獲利點**：
    *   **硬體溢價**：Mac Mini 硬體成本 + 軟體授權費（一次性高毛利）。
    *   **維護訂閱**：每年的軟體更新與遠端支援費（持續性收入）。

# 結論：從混亂到有序

這場 AI轉型的成功關鍵，不在於你用了多強的模型，而在於：

1.  **心法**：區分雜訊與結構，建立高品質的知識資產。
2.  **劍法**：透過 Hub & Spoke 平台確保資料主權。
3.  **心靈**：透過 DIKW 可視化，讓老闆看見「企業大腦」的成長，產生持續投入的動力。

**建議行動：** 優先開發「DIKW 星系儀表板」的原型 (Prototype)，即使只有單一框架 (如 SWOT)，也能展現巨大的商業說服力。
